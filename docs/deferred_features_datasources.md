## 個別所感

### 4. グレード/種目/ラウンド — 最優先（低コスト・高効果）
**結論: 追加の通信ゼロで取得できる。** すでに毎回取得しているページに埋まっている。
- レース一覧HTML（`gamboo_schedule.parse_race_numbers` が舐めている同じページ）に
  `<span class="name">ガールズ予選１</span>` があり、**種目＋ラウンド**（予選/準決/決勝/一次予選…）が判る。
- オッズページ `<title>`（`gamboo_racecard`/`dataset` が既にfetch）に「平塚ＧＰ … 初日」形式で
  **グレード（GP/GI/GII/GIII/F1/F2）・開催名・開催日目**が入る。開催日目は疲労指数(#6)にも流用可。
- schema には `races.grade` / `race_name` 列が既にありパイプライン側で埋めるだけ。
- 効果: レースタイプ分類（軸堅/標準/混戦）とバケット分割に効く見込み。決勝は堅く予選は荒れる等、
  ラウンドはガールズの荒れ度と相関が強いと言われる中核メタ情報。→ **次に着手すべき筆頭。**

### 5. 気象（バンク×風）
- 「逃げ有利/不利は風向×バンク直線で決まる」は競輪定説。バンク周長(既取得)と掛けると意味を持つ。
- **最小実装**: まず**GambooBET結果ページの `天候`・`風速` を既存 `gamboo_result` パースに1行追加**（無料・
  取得済ページ・工数小）。これでバックテスト時に「風速帯×バンク×脚質」のROI差を検証できる。
  ただし**風向が無い**ため「向かい風/追い風」判定は不可、風速の大小のみ。
- 風向まで欲しい／予測時(as-of)に使いたい場合のみ **気象庁 JMA API**（AMeDAS実測10分値＋予報JSON、
  無料・キー不要・利用制限緩め）で会場最寄アメダス点の 風向/風速/気温 を紐付ける（工数中）。
- 推奨順序: 先に無料の結果ページ風速で**シグナルの有無を検証** → 効くと分かってから JMA で風向を足す。
  費用対効果不明な段階でAPI連携を先に作らない。

### 6. 疲労指数
**新規ソース不要。** `days_since_last`（前走からの間隔）と `meet_results`（今場所の走行日付・R・着順）が
既にあり、「今場所の通算走行数」「連日出走か」「中○日」を特徴化できる。#4で取れる開催日目(初日/2日目…)
を足せば精度が上がる。→ 特徴量実装の話であってデータ取得課題ではない。

### 1・2・3b. 通過順位・包まれ率・バック順位別
**構造化データはどの許可ソースにも存在しない。** 隊列図はnetkeirin（規約でスクレイピング禁止＝不可）
のみ。GambooBET/オッズパーク/KEIRIN.JP公式は S/H/B の**集計回数**までで、レース内の周回別位置は
**映像**でしか提供されない。取得するなら (a) `勝敗因`コメントのNLP抽出（「後方」「前受け」「番手」「大外」
等 → 位置ラベル化、ノイズ多・工数大）か (b) 映像OCR（現実的でない）。**当面は見送り推奨。**

## 推奨着手順（費用対効果）

1. **【小・高】グレード/種目/ラウンドのパース追加**（#4）— 通信ゼロ、race-type/バケットに直結。最優先。
2. **【小・中】上がり順位の導出 ＋ 疲労指数の特徴化**（#3a, #6）— 新規取得ゼロ、既存カラムから計算。
3. **【小・中】GambooBET結果の天候/風速パース**（#5前半）— 無料、バックテストで風シグナル検証用。
4. **【中・?】JMA APIで風向/気温**（#5後半）— 3でシグナルが出た場合のみ着手。
5. **【大・低】通過順位/包まれ率（勝敗因NLP等）**（#1,2,3b）— 効果不確実かつ高コスト。長期課題として保留。

## 規約メモ
- netkeirin(keirin.netkeiba.com): スクレイピング禁止＋自動IP-BAN。隊列図があっても**使わない**（CLAUDE.md方針）。
- GambooBET / 競輪ステーション(/keirindb/) / オッズパーク: 既に利用中の主系。1秒以上間隔遵守。
- 気象庁JMA: 公開データ・API。出典明記で利用可（商用も配布形態次第、要規約確認）。
